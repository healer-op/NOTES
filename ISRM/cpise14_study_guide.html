Q1a 4 Marks
5 Components of an Information System — which are most impacted by Computer Security?
Mnemonic
"SHIPS sail through cyber storms"
S — Software  |  H — Hardware  |  I — Information/Data  |  P — People  |  S — Systems/Networks/Procedures
Definition (Stallings)

An Information System is a set of interrelated components that collect, process, store, and distribute information to support decision-making and control in an organization.

The 5 Components — Explained Simply
  • Hardware — All physical devices: servers, computers, routers, hard drives. Security impact: theft, physical damage, hardware keyloggers. Example: Stealing a laptop = instant data breach.
  • Software — Programs, OS, applications. Most directly attacked! Malware, ransomware, buffer overflow exploits target software. Example: WannaCry attacked a Windows OS vulnerability.
  • Data / Information — Raw facts stored in the system. Prime target of every attacker — they want YOUR data. Confidentiality, Integrity, Availability (CIA triad) all apply here. Example: Stealing a customer database.
  • People — Users, admins, operators. Called the "weakest link" in security. Social engineering, phishing, insider threats exploit people. Example: Employee clicking a phishing email = breach entry point.
  • Networks / Procedures — Communication links + rules for operating the system. Attacks: DoS, eavesdropping, man-in-the-middle. Example: Attacker sniffing unencrypted Wi-Fi traffic.
Most Impacted by Computer Security

1. Software — Most exploited. Every virus, malware, ransomware targets software vulnerabilities.
2. Data — Most valuable. The entire goal of most attacks is to steal or corrupt data.
3. Networks — Most exposed. DoS, MITM, eavesdropping are the most common modern attacks.
People and Hardware are impacted too, but Software + Data + Networks face the highest attack frequency.

Exam Note

Your teacher's version (Hardware, Software, Data, People, Procedures) is correct for the "5 components" part. For the "security impact" part, additionally mention Networks and highlight Software + Data as prime targets. Write both — you get full marks!

Q1b 4 Marks
What is the Security Development Life Cycle (SDLC)? Who leads the process?
Mnemonic — Phases in order
"Train Really, Design It, Verify Results"
Training → Requirements → Design → Implementation → Verification → Release → Response
Definition

The Security Development Life Cycle (SDL) is a software development process that adds security activities at every phase of development to reduce vulnerabilities in software before it is released.

7 Phases — Easy Explanation
  • Phase 1 — Training: Teach all developers about secure coding, threats, and best practices BEFORE they write a single line of code.
  • Phase 2 — Requirements: Define WHAT security the software needs. E.g., "must use AES-256 encryption for passwords."
  • Phase 3 — Design: Create threat models. Ask "how can an attacker break this?" and design to prevent it. Use attack surface analysis.
  • Phase 4 — Implementation: Write code following secure coding rules. Use static analysis tools to find bugs automatically.
  • Phase 5 — Verification: Test the software for security: penetration testing, fuzzing (sending random bad inputs), code review.
  • Phase 6 — Release: Final security check before launch. Prepare an incident response plan in case something goes wrong after launch.
  • Phase 7 — Response: After launch — monitor for issues, release patches for found vulnerabilities, handle security incidents quickly.
Who Leads the Process?

The CISO (Chief Information Security Officer) leads at the organizational level. Within the dev team, a Security Champion (a developer trained in security) leads day-to-day. Microsoft introduced this model in 2004 — it became the industry standard. NIST calls this the "Secure Software Development Framework (SSDF)" in SP 800-218.

Ref: Stallings Ch.1 Microsoft SDL 2004 NIST SP 800-218 (SSDF)
Q1c 4 Marks
Why is methodology important in Information Security? How does it improve the process?
Mnemonic
"CRISP methodology keeps security fresh"
C — Consistency  |  R — Repeatability  |  I — Identifies gaps  |  S — Standardization  |  P — Prioritization
What is a Methodology? (Simple Definition)

A methodology is a structured, step-by-step approach to solving a problem. In information security, it means following a defined process to identify, assess, and fix security issues — instead of doing things randomly or based on guesswork.

Why is it Important? — 5 Key Reasons
  • Consistency: Every system in the organization gets the same security level. No system is left unprotected because someone "forgot."
  • Repeatability: The process can be audited, repeated, and improved every time. Like a recipe — same steps = reliable result every time.
  • Identifies Gaps: A structured checklist finds vulnerabilities that random checks would miss. Nothing slips through the cracks.
  • Legal Compliance: Following NIST or ISO 27001 methodology proves to regulators you follow the law (IT Act 2000, GDPR). Avoids fines.
  • Resource Prioritization: Limited budget? Methodology tells you which risks to fix FIRST based on impact — not guesswork.
How it Improves the Process

Before methodology: Security is ad-hoc, reactive, inconsistent — patch one problem, miss ten others.
After methodology: Proactive, structured, auditable. The NIST CSF 2.0 methodology is: Govern → Identify → Protect → Detect → Respond → Recover. Each function has defined outcomes and measurable metrics — improvement is now trackable.

Real-World Example

A hospital without methodology patches whatever their vendor mentions that week. With NIST RMF methodology: they identify all assets, classify data (patient records = HIGH impact), apply controls based on risk level, and review every 6 months. Result: structured, auditable, and compliant security posture.

Ref: Stallings Ch.1 NIST CSF 2.0 NIST SP 800-37 Rev.2

Q2a 4 Marks
4 Common Threats: IP Compromise · Espionage · Sabotage · Software Attacks
Mnemonic
"Intelligent Enemies Sabotage Systems" = I·E·S·S
Intellectual Property · Espionage · Sabotage · Software Attacks
  • (i) Compromises to Intellectual Property (IP): Stealing trade secrets, source code, patents, or copyrighted material without permission. The attacker gains unfair competitive advantage. Countermeasures: DRM (Digital Rights Management), access control, NDAs, watermarking. Example: Rival company hacks your database to steal your product design.
  • (ii) Deliberate Acts of Espionage: Secretly monitoring or accessing systems to gather confidential information for competitive or political purposes. Countermeasures: End-to-end encryption, network monitoring, IDS/IPS. Example: Nation-state hackers spying on DRDO/ISRO research servers.
  • (iii) Deliberate Acts of Sabotage or Vandalism: Intentionally damaging or disrupting systems to cause operational harm. Can be physical (breaking hardware) or digital (deleting data, defacing websites). Countermeasures: Physical security, backups, intrusion prevention systems. Example: Stuxnet worm (2010) destroyed Iranian nuclear centrifuges.
  • (iv) Deliberate Software Attacks: Using malicious programs to damage, disrupt, or gain unauthorized access. Includes viruses, worms, ransomware, Trojans, spyware. Countermeasures: Antivirus software, regular patching, sandboxing, user training. Example: WannaCry ransomware (2017) encrypted files in 150 countries, demanded Bitcoin ransom.

Q2b 4 Marks
Define Risk Management + Roles of 3 Communities of Interest
Definition (Stallings — Exact Words)

"Risk Management is the process of identifying risk, assessing risk, and taking steps to reduce risk to an acceptable level." — Stallings, Computer Security: Principles and Practice

Mnemonic — 3 Communities
"IBM manages all risk"
I — IT Security community  |  B — Business/Functional community  |  M — Management community
  • Community 1 — IT Security: The technical experts who implement and operate security controls. They monitor for threats, respond to incidents, manage firewalls, IDS, and encryption. Role: Identify risks and implement technical controls. Know WHAT to protect and HOW.
  • Community 2 — Business / Functional: The departments that own and use business assets (Finance, HR, Operations). They understand the business value of information and define what risk level is acceptable. Role: Define acceptable risk thresholds and fund security investments.
  • Community 3 — Management / Executive: CEOs, Board of Directors, CISOs. They set overall direction, allocate budget, sign off on policies, and are legally accountable for security failures. Role: Set strategy, approve risk decisions, ensure regulatory compliance.
Simple Analogy

Think of a hospital: IT Security = doctors who diagnose and treat; Business = hospital departments that define patient safety needs; Management = hospital board that sets policy and budget. All three must work together for patient (data) safety.

Q2c 4 Marks
3 Main Risk Control Strategies with their Safeguards
Mnemonic
"ATA — Always Think About risk"
Avoidance (Apply safeguards) · Transference · Acceptance (+ Mitigation)
  • Strategy 1 — Avoidance (Apply Safeguards): Add security controls that eliminate or reduce the risk to an acceptable level. Best when: cost of control < cost of the risk. Safeguards: firewalls, AES-256 encryption, MFA (multi-factor authentication), access control lists, regular patching, user training. Example: Installing a firewall to block unauthorized network access.
  • Strategy 2 — Transference: Move the financial or operational responsibility to another party. Best when: risk cannot be eliminated but someone else can handle the cost. Safeguards: Cyber insurance, outsourcing to MSSP (Managed Security Service Provider), contractual liability clauses. Example: A hospital buys cyber insurance — if a breach occurs, insurance pays the damages.
  • Strategy 3 — Mitigation (Reduce Impact): Cannot eliminate the risk, but reduces damage WHEN it happens. Three sub-plans: (a) Incident Response Plan — immediate actions when attack happens. (b) Disaster Recovery Plan (DRP) — restore IT systems quickly. (c) Business Continuity Plan (BCP) — keep business running during recovery.
4th Strategy — Acceptance

Accept the risk when: cost to fix it is MORE than the value of the asset being protected. This must be a deliberate, documented management decision — not laziness or negligence. NIST requires formal sign-off by the Authorizing Official (AO).

Ref: Stallings Ch.3 NIST SP 800-39

4 Ways to Categorize Risk Control + Flowchart of Risk Control Cycle
Mnemonic
"PDCA — Police Detect Crime Always"
Preventive · Detective · Corrective · Administrative
  • 1. Preventive Controls: Stop the threat BEFORE it happens. Goal: block the attacker at the door. Examples: Firewall (blocks unauthorized traffic), access control (only authorized users enter), encryption (makes stolen data unreadable), security training (stops users from making mistakes).
  • 2. Detective Controls: Identify and ALERT when an attack is happening or has happened. Goal: catch the attacker in action. Examples: Intrusion Detection System (IDS), CCTV cameras, audit logs, SIEM, anti-virus scanning.
  • 3. Corrective Controls: Reduce damage AFTER the attack and restore normal operations. Goal: fix what the attacker broke. Examples: Data backups, patch management, incident response teams, system reimaging.
  • 4. Administrative Controls: Policies, rules, laws, and guidelines that govern how security is managed. Goal: create a framework of rules. Examples: Security policies, NIST frameworks, ISO 27001 compliance, employee acceptable-use policies, background checks.

Q3b 4 Marks
Mathematics of Cost-Benefit Analysis for Risk Control
Mnemonic — Key Terms
"AVoid Every Annual Loss" = AV · EF · SLE · ARO · ALE
Asset Value · Exposure Factor · Single Loss Expectancy · Annual Rate of Occurrence · Annual Loss Expectancy
Step-by-Step Formulas
Step 1: Asset Value (AV)
  = How much is this asset worth to the organization?
  Example: Customer database = ₹10,00,000

Step 2: Exposure Factor (EF)
  = What % of the asset is lost if threat occurs? (0 to 1)
  Example: Ransomware encrypts 30% of data → EF = 0.30

Step 3: Single Loss Expectancy (SLE)
  SLE = AV × EF
  SLE = ₹10,00,000 × 0.30 = ₹3,00,000
  (= How much you lose in ONE attack incident)

Step 4: Annual Rate of Occurrence (ARO)
  = How many times does this attack happen per year?
  Example: Ransomware hits 2 times/year → ARO = 2

Step 5: Annual Loss Expectancy (ALE)
  ALE = SLE × ARO
  ALE = ₹3,00,000 × 2 = ₹6,00,000
  (= Total expected loss per year WITHOUT any control)

Step 6: Cost-Benefit Analysis (CBA)
  CBA = ALE(before control) − ALE(after control) − Annual Cost of Control

  Example: ALE before = ₹6L, ALE after = ₹1L, Control cost = ₹2L/year
  CBA = ₹6L − ₹1L − ₹2L = ₹3,00,000 ← POSITIVE → Implement it!
Decision Rule — Easy to Remember

If CBA is POSITIVE → implement the control (you save more than you spend).
If CBA is NEGATIVE → control costs more than the risk itself → accept risk or find cheaper control.
Think of car insurance: if premium (cost) < expected accident cost → buy the insurance!

Ref: Stallings Ch.3 NIST SP 800-30 Rev.1 Appendix G
Q3c 4 Marks
Define Risk Mathematically + Compute Risk for the Given Problem
Mathematical Definition of Risk

Risk = Likelihood of threat × Value of information asset × Vulnerability level

In simpler form: Risk = Threat × Asset Value × Vulnerability

Quantitatively: Risk is measured as ALE = SLE × ARO (expected annual financial loss).

Solving the Given Problem — Step by Step
Given Information:
  Asset Value (AV)  = 100
  Data accuracy     = 80%  (multiply final answer by 0.8)

Vulnerability 1:
  Likelihood                    = 0.5
  Current control covers        = 60% of the risk
  Residual likelihood           = 0.5 × (1 − 0.60)
                                = 0.5 × 0.40
                                = 0.20
  Risk from V1                  = 0.20 × 100 = 20

Vulnerability 2:
  Likelihood                    = 0.2
  Current control covers        = 70% of the risk
  Residual likelihood           = 0.2 × (1 − 0.70)
                                = 0.2 × 0.30
                                = 0.06
  Risk from V2                  = 0.06 × 100 = 6

Total Risk (before accuracy):
  = Risk(V1) + Risk(V2) = 20 + 6 = 26

Adjusted for 80% data accuracy:
  = 26 × 0.80 = 20.8

∴ Total Residual Risk ≈ 20.8 (out of 100)
What This Answer Means

The organization still faces a residual risk of approximately 20.8% on this asset even after existing controls are applied. Since this may exceed the organization's risk appetite, additional controls should be evaluated using CBA. The 80% accuracy factor reflects real-world uncertainty — risk assessments always have some built-in uncertainty.

Q4a 4 Marks
Discuss Enterprise Information Security Policy (EISP)
Mnemonic — EISP Components
"OSCAR runs the enterprise security"
Overview · Scope · Compliance · Authority/Roles · References to sub-policies
Definition

The Enterprise Information Security Policy (EISP) is the highest-level security policy document in an organization. It sets the overall philosophy, direction, and framework for all information security activities. Also called the "Security Program Policy" or "Master Security Policy."

  • Overview / Statement of Purpose: WHY the organization cares about security. Sets the security philosophy at board level. Example: "XYZ Bank is committed to protecting customer financial data from unauthorized access."
  • Scope: WHO and WHAT is covered. All employees, contractors, third-party vendors. All systems: cloud, on-premise, mobile. No one is exempt from this policy.
  • Authority & Roles: WHO is responsible? CISO owns the policy. IT department implements it. Employees must follow it. Management approves it. Clear chain of accountability established.
  • Compliance Requirements: Legal and regulatory obligations the organization must meet: IT Act 2000 (India), GDPR (Europe), PCI-DSS (payments), HIPAA (healthcare). Non-compliance = fines + legal action.
  • References: Points to lower-level policies: Issue-Specific Policies (email use, BYOD) and System-Specific Policies (firewall rules, server config). EISP is the umbrella; these are the details.
Example EISP Statement

"ABC Corporation is committed to ensuring the confidentiality, integrity, and availability of all its information assets. All employees and contractors are required to comply with this policy. The CISO is responsible for maintaining and enforcing this policy."

Exam Note — Important!

EISP is the umbrella policy. It does NOT go into technical details — it says WHAT the organization will do about security. The HOW is handled by Issue-Specific and System-Specific policies below it. Think of EISP as the Constitution — everything else flows from it.

Ref: Stallings Ch.4 "Security Policies" NIST SP 800-12 Rev.1 Ch.5

Q4c 4 Marks
Risk Management Process in Project Management
Mnemonic — PMBOK Risk Steps
"Please Identify Quality Quantities, Plan Responses, Monitor"
Plan → Identify → Qualitative Analysis → Quantitative Analysis → Plan Responses → Monitor & Control
  • Step 1 — Plan Risk Management: Decide HOW you will handle risk in this project. Define methodology, roles, tools, and budget for risk activities. Output: Risk Management Plan document.
  • Step 2 — Identify Risks: List ALL possible things that could go wrong. Use brainstorming, checklists, SWOT analysis, expert interviews. Output: Risk Register (a log of all identified risks).
  • Step 3 — Qualitative Risk Analysis: Prioritize risks using a Probability × Impact Matrix (High / Medium / Low). Fast, no math needed. Helps decide which risks need deeper analysis first.
  • Step 4 — Quantitative Risk Analysis: For high-priority risks, calculate actual financial impact. Methods: Monte Carlo simulation, Expected Monetary Value (EMV), decision trees. Output: Numerical risk scores.
  • Step 5 — Plan Risk Responses: Decide what to DO about each risk. Options: Avoid (change plan), Transfer (insurance), Mitigate (reduce likelihood/impact), Accept (document and move on).
  • Step 6 — Monitor & Control Risks: Throughout the entire project, track risks, check if responses are working, and identify new risks. This is never a one-time activity — it's continuous.
Example — Cloud Migration Project

Risk identified: "Data breach during migration." Qualitative: High probability × High impact → Priority 1. Quantitative: Estimated loss = ₹50 Lakhs. Response: Mitigate — use encrypted transfer (TLS 1.3), MFA for all access, penetration testing before go-live. Monitor: Weekly security review meetings during the migration window.

Ref: PMBOK 7th Edition (PMI) Stallings Ch.3 NIST SP 800-30

Q5a 4 Marks
Case Study: BHIM App and Google Pay in the Context of Risk
Background

BHIM (Bharat Interface for Money) and Google Pay are UPI-based digital payment systems used by millions in India. In 2020, cybersecurity firm vpnMentor discovered that 7.26 million BHIM users' sensitive data was exposed due to a misconfigured AWS S3 cloud storage bucket — left publicly accessible with no password protection.

  • Risk 1 — Misconfiguration (BHIM Breach): AWS S3 bucket was mistakenly set to "public." Exposed: Aadhaar numbers, PAN cards, bank account details, UPI IDs of 7.26M users. NIST Category: Protect — Configuration Management failure. Countermeasure: Regular cloud security audits, CSPM tools (Cloud Security Posture Management), principle of least privilege.
  • Risk 2 — Data Privacy Violation: Aadhaar-linked biometric and financial data exposed. Risk: Identity theft, financial fraud, account takeover. Countermeasure: Data minimization (collect only what is needed), encryption at rest (AES-256), tokenization of all sensitive fields.
  • Risk 3 — Phishing & Fake Apps: Fraudsters create fake BHIM/GPay apps identical to real ones. Users enter UPI PIN → attacker captures credentials. Countermeasure: App signing with certificates, Google Play Protect, NPCI user awareness campaigns.
  • Risk 4 — SIM Swap / OTP Interception: Attacker convinces telecom to transfer victim's number to attacker's SIM → now receives all OTPs. Countermeasure: Device binding (UPI app linked to one specific device), biometric authentication, anomaly detection (unusual transaction location/amount triggers alert).
Google Pay's Security Advantage

Google Pay uses tokenization — actual card/bank numbers are never stored or transmitted. A one-time token is generated per transaction. Additionally uses Google's global threat intelligence and machine learning to flag suspicious transactions in real-time. This is why GPay has had significantly fewer breaches than BHIM historically.

NIST CSF 2.0 Applied to Both

Identify: Map all assets (user data, UPI IDs, servers) → Protect: Encrypt data, use MFA, train staff → Detect: Monitor for unusual transactions, real-time anomaly alerts → Respond: Freeze compromised accounts instantly → Recover: Notify users, restore access, compensate losses.

Ref: NIST CSF 2.0 RBI Cybersecurity Framework 2021 vpnMentor BHIM Report 2020
Q5b 4 Marks
DoS vs DDoS — Which is More Dangerous and Why?
Mnemonic
"DDoS is Deadly because it's Distributed from a Dark army"
Single attacker (DoS) vs Millions of bots/zombies (DDoS) — scale makes DDoS far more devastating and nearly impossible to stop
Definitions

DoS (Denial of Service): A single attacker sends massive amounts of traffic or requests to a target server to overwhelm it, making it unavailable to legitimate users.

DDoS (Distributed Denial of Service): Thousands or millions of compromised machines (called a botnet or zombie army) simultaneously attack a target from different locations worldwide.

DoS Attack
  • Single attacker, single source IP
  • Relatively easy to block (block that one IP)
  • Limited bandwidth — one machine
  • Attacker is easily traceable
  • Impact: Moderate, usually short-lived
  • Example: SYN flood from one PC
DDoS Attack
  • Thousands/millions of machines (botnet)
  • Cannot block — IPs change constantly
  • Massive bandwidth: up to 1–3 Tbps
  • Attacker hides behind innocent victims
  • Impact: Catastrophic, long-lasting
  • Example: Mirai Botnet, 2016 (1.2 Tbps)
Why DDoS is More Dangerous — 4 Reasons

1. SCALE: Mirai Botnet (2016) generated 1.2 Tbps — took down Dyn DNS, making Twitter, Netflix, Amazon, and Reddit unreachable for hours across the USA.
2. ANONYMITY: Real attacker controls bots remotely and is nearly untraceable — they hide behind innocent hacked devices.
3. IMPOSSIBLE TO SIMPLY BLOCK: Cannot block millions of different IPs. Traditional firewalls completely fail against this scale.
4. ECONOMIC DAMAGE: Average DDoS attack costs a company $50,000–$2 million per hour in lost revenue and recovery costs.

Countermeasures

For DoS: Block source IP at firewall, rate limiting on server.
For DDoS: CDN-based scrubbing centers (Cloudflare, Akamai), Anycast traffic diffusion, ISP-level traffic filtering, BGP blackholing, dedicated cloud DDoS protection services.

Ref: Stallings Ch.7 "Denial of Service" NIST SP 800-61 Rev.2 Mirai Botnet Case 2016
Q5c 4 Marks
Discuss: (i) Man-in-the-Middle Attack   (ii) Risk Appetite   (iii) Residual Risk
Mnemonic — Remember All 3
"MAN with Appetite leaves Residue"
MITM attack · Risk Appetite (how much you'll accept) · Residual Risk (what's left after controls)
(i) Man-in-the-Middle (MITM) Attack — Definition

A MITM attack is when an attacker secretly positions himself between two communicating parties — intercepting, reading, and sometimes modifying their messages — without either party knowing the communication has been compromised.

  • How it works: Alice sends message to Bob. Attacker intercepts it, reads it, may modify it, then forwards to Bob. Both Alice and Bob think they're communicating directly with each other.
  • Types: ARP Spoofing (on local network), SSL Stripping (downgrades HTTPS → HTTP), DNS Spoofing (redirects to fake website), Rogue Wi-Fi (fake hotspot at coffee shop/airport).
  • Threats caused: Confidentiality breach (attacker reads your data), Integrity breach (attacker modifies messages), Authentication bypass.
  • Real example: You connect to "Free_Cafe_WiFi" — it's the attacker's rogue hotspot. They intercept your banking session and steal your login credentials.
  • Countermeasures: Always use HTTPS/TLS, use VPN on public Wi-Fi, certificate pinning in apps, mutual authentication (both sides verify identity).
(ii) Risk Appetite — Definition

Risk Appetite is the amount and type of risk an organization is willing to accept in pursuit of its business objectives. It is set by senior management or the Board of Directors and documented in the Risk Management Strategy.

  • High risk appetite: A startup that prioritizes fast growth over security. Willing to accept some data breach risk to ship features quickly. Less investment in controls.
  • Low risk appetite: A hospital or bank — near-zero tolerance for data breaches because patient/customer lives and money are at stake. Heavy investment in controls.
  • How it's used: Every security decision is checked against risk appetite. If residual risk > risk appetite → must add more controls or escalate decision to management.
(iii) Residual Risk — Definition

Residual Risk is the risk that REMAINS after all security controls have been applied. It is impossible to eliminate 100% of risk — what's left over is residual risk.

Formula: Residual Risk = Total/Inherent Risk − Risk covered by controls

  • Example: A firewall blocks 95% of attacks. The remaining 5% chance of a successful attack = residual risk.
  • What to do with it: Compare residual risk with risk appetite. If residual risk < risk appetite → acceptable, formally document and sign off. If residual risk > risk appetite → add more controls.
  • NIST process: The Authorizing Official (AO) formally reviews and accepts residual risk before a system goes live (NIST SP 800-37 Rev.2).
  • Key insight: Residual risk never reaches zero. Even after every possible control, there is always some unknown or uncontrollable risk remaining — this is a fundamental truth of security.
How All 3 Connect Together

A MITM attack is a threat. An organization's Risk Appetite determines how much chance of MITM they can tolerate. After applying controls (TLS, VPN, certificate pinning), the remaining chance of a MITM attack succeeding = Residual Risk. If residual risk exceeds risk appetite, stronger controls must be implemented.

Ref: Stallings Ch.6 (MITM) NIST SP 800-39 §2.3 (Risk Appetite) NIST SP 800-37 Rev.2 (Residual Risk)
