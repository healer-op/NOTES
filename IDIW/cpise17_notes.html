CO1 — Explain intrusion detection methods
1.1 — What is an Intrusion? What is an IDS?
Mnemonic
"IDS = Intelligent Door Security — watches who comes in"
Monitors → Analyses → Alerts → (IPS also Blocks)
Definitions (Stallings)

Intrusion: Any unauthorized access, use, modification, or denial of service on a computer system or network by a person (insider or outsider) or an automated attack tool.

Intrusion Detection System (IDS): Software and/or hardware that monitors network traffic or host activity and detects possible intrusions, generating alerts for security administrators.

Intrusion Prevention System (IPS): An IDS that can also automatically BLOCK detected attacks in real-time — not just alert.

Who are intruders?
  • Masqueraders: External attackers who gain unauthorized access by using a legitimate user's credentials (stolen passwords).
  • Misfeasors (Insider threats): Legitimate users who abuse their authorized access — e.g., an employee copying company data before resigning.
  • Clandestine users: Attackers who gain superuser/admin access and use it to cover their tracks and evade audit controls.
IDS vs IPS — Key Difference

IDS (Detection only): Monitors traffic, generates alerts, does NOT block. Like a security camera — it records everything but cannot physically stop a thief.
IPS (Detection + Prevention): Monitors AND automatically blocks malicious traffic. Like a security guard with a locked door — can physically stop the threat.
Modern systems are called IDPS (Intrusion Detection and Prevention System) — they do both.

1.2 — Signature-Based (Misuse) Detection
Mnemonic
"SIGDET — See If it matches, Generate alert, Defend, Else let Through"
Capture → Match Signature DB → Alert if match → Pass if no match
Definition

Signature-based detection maintains a database of known attack patterns (signatures). Observed traffic/events are compared against this database. A match = alert. Also called Knowledge-based or Misuse Detection.

  • Signature Database: Contains thousands of patterns — specific byte sequences, protocol flags, regex patterns. Example: Snort has 50,000+ community signatures. Must be updated frequently (like antivirus).
  • Exact Match: Byte-by-byte comparison. Very fast but rigid — a single bit change in attack code breaks the match. Polymorphic malware evades this.
  • Heuristic Match: Uses flexible rules — partially matches similar patterns. More resilient to small variations. Slightly higher false positives.
  • State-based Match: Tracks multi-step attacks that unfold over several packets/sessions. Example: detecting a port scan followed by a specific exploit attempt — both needed for alert.
  • Snort Rule Syntax: alert tcp any any → 192.168.1.0/24 80 (content:"/etc/passwd"; msg:"Directory Traversal"; sid:1001;)
✅ Advantages
  • Very low false positives
  • Fast processing (simple matching)
  • Easy to understand and audit
  • Precise — tells you exactly what attack type
❌ Disadvantages
  • Cannot detect zero-day attacks
  • Requires constant signature updates
  • Polymorphic malware evades it
  • Fails against encrypted attacks
Real Example

Snort IDS detects SQL injection: a web request containing ' OR '1'='1 matches a known SQL injection signature → Snort fires alert "SQL Injection Attempt" and logs the attacker's IP. The admin is notified within milliseconds.

1.3 — Anomaly-Based (Statistical) Detection
Mnemonic
"ALIEN spotted — Anomaly = Looks weird compared to normal"
Train on normal → Monitor live → Detect deviation → Alert
Definition

Anomaly-based detection builds a statistical model of normal system/user behaviour (baseline). Any significant deviation from this baseline triggers an alert. Can detect unknown (zero-day) attacks since it doesn't need prior knowledge of the attack.

  • Baseline Training Phase: System monitors "clean" traffic for days/weeks to learn normal patterns — average bandwidth, login times, typical commands used, normal packet sizes. This profile is the baseline.
  • Statistical Models used: Mean and Standard Deviation (flag if value exceeds μ ± 3σ), Markov Models (sequence analysis), Neural Networks (deep learning-based modern systems), Clustering algorithms (K-means).
  • Types of anomalies detected: (a) Volume anomaly: 10GB download at 3AM. (b) Protocol anomaly: malformed packet headers. (c) Behaviour anomaly: employee accessing 1000 files in 5 minutes (likely data exfiltration).
  • Self-Learning: Modern anomaly IDS uses Machine Learning (ML) — the model continuously updates as "normal" evolves. Avoids manual re-tuning.
6 Limitations — Must Know for Exam!

HITFAC:
High false positives — unusual but legitimate activity triggers alerts
Insider threats missed — malicious but "normal-looking" behaviour
Training overhead — weeks of clean traffic needed
Fool the IDS — slow/gradual attacks stay within threshold (mimicry attack)
Adapting environments — normal changes constantly (new users, new apps)
Computational cost — continuous statistical analysis is CPU-heavy

1.4 — Specification-Based and Hybrid Detection
Mnemonic
"Spec = Strict rulebook; Hybrid = Best of both worlds"
Specification = whitelist of allowed behaviour · Hybrid = Signature + Anomaly combined
Specification-Based Detection

Defines the correct/expected behaviour of a system using formal specifications (like a whitelist). Any behaviour OUTSIDE the specification = alert. Unlike anomaly detection, specifications are written by humans — not learned statistically. Example: "A web server process should only access files in /var/www/ — any other file access = alert."

  • Advantage over Anomaly: Low false positives (because specifications are precise, not statistical). Can detect insider threats (even legitimate users violating spec).
  • Advantage over Signature: Can detect zero-day attacks (unknown attacks still violate spec).
  • Limitation: Very hard and expensive to write complete specifications for complex systems. Any unspecified legitimate behaviour = false positive.
  • Use cases: SCADA/ICS systems (power grids, water treatment — very predictable behaviour), network protocols, safety-critical embedded systems.
Hybrid Detection

Combines two or more detection methods — most commonly Signature + Anomaly. Uses signature detection for known attacks (fast, accurate) AND anomaly detection for unknown attacks (catches zero-days). Modern enterprise IDS systems (like Cisco Firepower, Palo Alto NGFW) are all hybrid.

2.1 — Introduction to Information Warfare (IW)
Mnemonic
"IW = Information as a Weapon and a Target"
Attacker uses information to fight AND fights to protect/destroy information systems
Definition

Information Warfare (IW) is the use of information and information technology as both a weapon and a target in conflicts. It involves attacking, defending, and exploiting information systems to gain competitive or strategic advantage over an adversary.

  • Offensive IW: Attacking enemy information systems — hacking, jamming, propaganda, cyber attacks on infrastructure.
  • Defensive IW: Protecting own information systems — firewalls, IDS, encryption, redundancy, deception (honeypots).
  • 3 Core Goals of IW: (a) Deny enemy information. (b) Exploit enemy's reliance on information. (c) Protect own information systems from attack.
  • Difference from traditional warfare: No physical presence needed. Anonymous attacker. Attacks cross international borders instantly. Civilian infrastructure is now a prime target.
Ref:Stallings "Network Security Essentials" Ch.1
2.2 — Taxonomy (Classification) of Information Warfare
Mnemonic — 7 Classes
"CHIN-EPI = Command, Hacker, Intelligence, Network, Economic, Psychological, Information"
7 recognized categories of Information Warfare — remember by CHIN-EPI
  • 1. Command and Control Warfare (C2W): Attacking the enemy's ability to command and control forces. Targets: communication systems, headquarters, command links. Goal: cut decision-makers off from their forces. Example: DDoS on a government's emergency communication system during a crisis. Military: GPS jamming of enemy drones.
  • 2. Intelligence-Based Warfare (IBW): Using technology to gather intelligence and deny the enemy the same. Includes: satellite imagery analysis, signals intelligence (SIGINT), cyber espionage, surveillance systems. Example: NSA PRISM surveillance programme — collecting metadata from internet companies globally.
  • 3. Electronic Warfare (EW): Using the electromagnetic spectrum as a weapon. Radar jamming, GPS spoofing, communications interception (SIGINT), electronic countermeasures (ECM). Example: Russia GPS spoofing near NATO waters — ships in Black Sea showed false GPS locations.
  • 4. Psychological Warfare (PSYWAR): Influencing, manipulating, or demoralizing enemy forces or civilian population through information. Tools: propaganda, fake news, social media manipulation. Example: Cambridge Analytica — used Facebook data to target psychological profiles with political ads to influence elections.
  • 5. Hacker Warfare: Using hacking skills to disrupt or destroy enemy information systems. Examples: Anonymous hacktivist group attacking government sites; state-sponsored hackers targeting military databases.
  • 6. Economic Information Warfare: Targeting financial systems, stock markets, banking infrastructure for economic destabilization. Example: NotPetya malware (2017) — initially disguised as ransomware, it permanently destroyed data at Maersk (shipping), FedEx, and others. $10 billion total damage.
  • 7. Cyber Warfare / Critical Infrastructure Attacks: Nation-state attacks on power grids, water systems, hospitals, financial networks. Example: Colonial Pipeline attack (2021) — ransomware shut down 45% of US East Coast fuel supply. Ukraine power grid attacks (2015, 2016) — first confirmed cyber attack to cause physical power outages.
Most Famous Example — Stuxnet (2010)

Stuxnet combined C2W (sabotaging Iranian nuclear command) + IBW (intelligence gathering on centrifuge specifications) + Cyber Warfare (physically destroying centrifuges through PLC software manipulation). It was the world's first documented cyber weapon designed to cause physical destruction. Joint US-Israel operation.

2.3 — Network-Based Attacks
Mnemonic
"SPAM-DR — Sniffing, Port scan, ARP poison, MITM, DoS, Replay"
6 primary network-based attack categories
  • Sniffing / Eavesdropping (Passive): Attacker places NIC in promiscuous mode to capture all network packets — even those not addressed to them. Unencrypted protocols (HTTP, FTP, Telnet) fully exposed. Tool: Wireshark. Countermeasure: Use HTTPS/TLS, SSH instead of Telnet.
  • Port Scanning (Reconnaissance): Systematically probing ports to find open services. Types: TCP Connect Scan (noisy), SYN Scan (stealthy — half-open), UDP Scan, NULL Scan. Tool: Nmap. Countermeasure: Firewall blocks unknown ports, IDS detects rapid port access.
  • ARP Poisoning (MITM Setup): ARP (Address Resolution Protocol) has no authentication. Attacker sends fake ARP replies: "IP 192.168.1.1 is at MAC AA:BB:CC..." — all network machines update their ARP cache. Traffic now flows through attacker. Enables MITM on LAN. Tool: arpspoof, Ettercap.
  • Man-in-the-Middle (MITM): Attacker intercepts and possibly modifies traffic between two parties. Enabled by: ARP poisoning, rogue Wi-Fi, DNS spoofing. Can read credentials, modify bank transfers, inject malicious content. Countermeasure: TLS, VPN, certificate pinning, HSTS.
  • IP Spoofing: Attacker forges source IP address in packets to impersonate another host, bypass IP-based authentication, or launch reflected DDoS. Countermeasure: Ingress filtering (ISPs block packets with spoofed IPs from their customers).
  • Replay Attack: Attacker captures a valid authentication packet (e.g., hashed login) and replays it later to gain access — without knowing the original password. Countermeasure: Timestamps, Nonces (one-time random values), session tokens.
Ref:Stallings Ch.18-19
2.4 — Wireless-Based Attacks
Mnemonic
"RIDE — Rogue AP, Injection, Deauth, Eavesdropping"
4 main wireless attack types
  • Passive Eavesdropping: Wi-Fi signals broadcast in all directions — anyone in range can capture packets using Wireshark + wireless card. WEP-encrypted networks are fully crackable. Countermeasure: WPA3 encryption.
  • Rogue Access Point / Evil Twin: Attacker sets up a Wi-Fi AP with the same SSID as a legitimate network. Sends deauthentication frames to disconnect users from real AP → users reconnect to attacker's AP → all traffic intercepted. Example: "Airport_Free_WiFi" vs real airport hotspot.
  • Deauthentication (Deauth) Attack: 802.11 management frames are unencrypted by default. Attacker sends fake deauth frames → victims forcibly disconnected. Used to: (a) force reconnection to rogue AP, (b) capture WPA2 4-way handshake for offline cracking, (c) DoS — keep deauthenticating = persistent disconnection. Fix: 802.11w (Management Frame Protection).
  • WEP Cracking: WEP uses RC4 with weak IV (Initialization Vector) — IVs repeat after ~5000 packets. Statistical analysis of repeated IVs reveals the key. Tool: Aircrack-ng. Can crack WEP in minutes with 50,000 packets. WEP is completely broken — should NEVER be used.
  • WPA2 Cracking: Capture 4-way handshake (during deauth attack) → run offline dictionary/brute-force attack against captured handshake. Tool: Aircrack-ng + wordlist. Only works against weak passwords. WPA2 itself is secure with strong passwords. WPA3 uses SAE — immune to offline dictionary attacks.
  • Karma/SSID Probe Attack: Devices broadcast "Probe Requests" for previously connected networks ("looking for Home_WiFi"). Attacker responds to all probes, impersonating every network the device ever connected to — device auto-connects. Completely transparent to user.
Attack Flow — Evil Twin

1. Attacker sets up AP: SSID = "Coffee_Shop_WiFi" (identical to real) · 2. Sends deauth to users on real AP · 3. Users reconnect to Evil Twin · 4. Attacker runs SSLstrip to downgrade HTTPS to HTTP · 5. Captures usernames, passwords, session cookies · 6. User never knows — internet still works.

Ref:Stallings Ch.19 "Wireless Security"
2.5 — Phases of an Electronic Attack (Cyber Kill Chain)
Mnemonic
"Real Spies Scan Every Exploit, Then Cover" = RS·SEETC
Reconnaissance → Scanning → Enumeration → Exploitation → Escalation → Tracks Covered
  • Phase 1 — Reconnaissance: Passive: Google dorking, WHOIS, LinkedIn, Shodan. Active: DNS queries, traceroute. Goal: build a complete picture of the target without touching their systems. Footprinting the target.
  • Phase 2 — Scanning & Probing: Actively test open ports (Nmap), OS fingerprinting, service version detection, vulnerability scanning (Nessus, OpenVAS). This phase leaves traces in firewall/IDS logs.
  • Phase 3 — Enumeration: Extract specific info: usernames (null SMB session), shared resources, group memberships, routing tables, SNMP community strings. Deeper than scanning — requires service interaction.
  • Phase 4 — Exploitation / Gaining Access: Use discovered vulnerability to get a shell or execute code. Methods: exploit known CVE (Metasploit), social engineering (spear phishing), brute force credentials, web app attacks (SQLi, XSS, LFI). First foothold established.
  • Phase 5 — Privilege Escalation & Lateral Movement: Escalate from user to admin/root (sudo misconfig, kernel exploit, SUID binary). Move across network to reach valuable targets (database server, domain controller). Pivot through the network.
  • Phase 6 — Maintaining Access: Install backdoor (Netcat, reverse shell, RAT), rootkit (hides attacker presence), scheduled tasks for persistence. Goal: survive reboots and password changes.
  • Phase 7 — Covering Tracks: Clear logs (Event Viewer, /var/log/auth.log, bash_history), modify timestamps, delete evidence. Advanced attackers use log-tampering tools and live-off-the-land techniques to blend in.
Lockheed Martin Kill Chain (7 Steps)

Reconnaissance → Weaponization → Delivery → Exploitation → Installation → Command & Control (C2) → Actions on Objectives.
Defender's goal: Break the chain as early as possible. Best: detect at Reconnaissance. Worst: discover only at "Actions on Objectives" (data already stolen).

CO2 — Attack types and their mechanisms
3.1 — Denial of Service (DoS) and DDoS Attacks
Mnemonic — DoS Types
"SYN-PUS = SYN flood, Ping of death, UDP flood, Smurf"
4 classic DoS attacks — plus HTTP flood and Amplification for modern attacks
Definition

A Denial of Service (DoS) attack attempts to make a system unavailable to its legitimate users by overwhelming it with traffic or exploiting vulnerabilities to consume all resources. A DDoS uses thousands/millions of compromised machines simultaneously.

  • SYN Flood: Exploits TCP 3-way handshake. Attacker sends thousands of SYN packets with spoofed IPs. Server allocates resources for each half-open connection and waits for ACK (that never comes). Connection table fills up → legitimate connections refused. Countermeasure: SYN Cookies (server doesn't allocate resources until handshake complete).
  • Ping of Death: Sends ICMP ping packet larger than 65,535 bytes (max allowed). Old OS cannot handle oversized packet reassembly → buffer overflow → system crash. Historic attack — all modern OS are patched. Still asked in exams!
  • UDP Flood: Sends massive UDP packets to random ports. Target checks each port for listening application, sends "ICMP Port Unreachable" for each → CPU and bandwidth exhausted. Simple to launch — no connection handshake needed.
  • Smurf Attack: Attacker pings network broadcast address with victim's spoofed IP. ALL hosts on network reply to victim → amplification attack. One small ping packet → hundreds of replies hit victim. Countermeasure: Disable directed broadcasts on routers.
  • HTTP Flood (Application Layer L7): Sends legitimate-looking HTTP GET/POST requests in massive volume. Bypasses network-level defences. Server must fully process each request. Slowloris: opens partial HTTP connections, holds them open indefinitely → exhausts server thread pool.
  • Amplification/Reflection Attack: Attacker sends small packet with victim's spoofed IP to public DNS/NTP server. Server sends LARGE response to victim. Amplification factor: DNS=28-54x, NTP=556x, Memcached=51,000x. Countermeasure: BCP38 (ingress filtering), disable open resolvers.
  • DDoS via Botnet: Thousands of compromised IoT devices, PCs, servers — all controlled by attacker's C2 server. Scale: 1-3 Tbps. Mirai Botnet (2016): hacked IoT cameras/routers using default passwords → generated 1.2 Tbps → took down Dyn DNS → Twitter, Netflix, Reddit unreachable.
3 Layers of DoS

Volumetric (L3/L4): Saturate bandwidth — UDP flood, ICMP flood, Amplification
Protocol (L4): Exhaust connection tables — SYN flood, Smurf
Application (L7): Exhaust server resources — HTTP flood, Slowloris
DDoS protection needs defence at all 3 layers simultaneously.

Ref:Stallings Ch.7 "Denial of Service"
3.2 — Buffer Overflow Attack
Mnemonic
"Buffer BREAKS the BOUNDARY, Return address REDIRECTED"
Overflow → overwrite return address → redirect execution → shellcode runs → attacker wins
Definition (Stallings)

A buffer overflow occurs when a program writes data beyond the boundaries of a fixed-length memory buffer, overwriting adjacent memory locations — including the function's return address on the stack. Attacker crafts malicious input to overwrite the return address, redirecting execution to attacker-controlled shellcode.

  • Why it happens: C/C++ functions like strcpy(), gets(), sprintf() do not check input length. If input is larger than the buffer, it overflows into adjacent stack memory.
  • Stack Layout (Normal): Function call pushes: [Local Variables / Buffer] → [Saved Frame Pointer (SFP)] → [Return Address (RET)] → [Function Arguments].
  • The Attack: Attacker provides input larger than buffer size. Overflows into SFP → overwrites RET with address of attacker's shellcode. When function returns → CPU jumps to attacker's shellcode instead of legitimate code.
  • Shellcode: Machine code embedded in the attacker's input. Typically opens a root shell (e.g., execve("/bin/sh")). Must be small (fits in buffer) and avoid null bytes (strcpy stops at null).
  • Heap Overflow: Same concept but in heap-allocated memory (malloc). Overwrites object pointers, vtable pointers, or heap metadata. Harder to exploit but also harder to defend.
  • Famous Exploits: Morris Worm (1988) — fingerd buffer overflow. MS03-026 (Blaster worm) — RPC buffer overflow. EternalBlue (WannaCry) — SMB protocol buffer overflow.
5 Countermeasures — Must Know

1. Stack Canaries: Compiler places a random "canary" value between buffer and return address. Before function returns, checks if canary was modified — if changed, abort (attacker's overflow is detected).
2. ASLR (Address Space Layout Randomization): Randomizes memory addresses each time program runs — attacker cannot predict where to jump.
3. DEP/NX Bit (Data Execution Prevention): Marks stack as non-executable. Shellcode in stack cannot run — CPU raises exception. Defeated by ROP (Return-Oriented Programming) attacks.
4. Safe Functions: Use strncpy() instead of strcpy(), fgets() instead of gets() — specify maximum input length.
5. Code Auditing & Fuzzing: Static analysis tools (Coverity, CodeQL), dynamic fuzzing (AFL — American Fuzzy Lop) to find overflow bugs before deployment.

Ref:Stallings Ch.10 "Buffer Overflow"
3.3 — Sniffing, Session Hijacking, Password Attacks
Mnemonic
"SSP = Sniff the wire, Steal the session, Punch the password"
3 related attack categories that chain together in real attacks
Sniffing

Passive capture of network packets passing through a network interface. The NIC is placed in promiscuous mode — captures ALL packets on segment, not just those addressed to it. Can capture cleartext credentials from HTTP, FTP, Telnet, POP3, SMTP.

  • Types: Passive sniffing (hub network — all traffic visible), Active sniffing (switched network — requires ARP poisoning to redirect traffic to sniffer).
  • What can be captured: Usernames and passwords (unencrypted), Email content, Web browsing history, Session cookies (enables session hijacking), VoIP calls.
  • Countermeasures: Encrypt all traffic (TLS/HTTPS/SSH/VPN). Use switches (not hubs). Monitor ARP table for poisoning. Use tools like XArp to detect ARP attacks.
Session Hijacking

Taking over an active, authenticated user session by stealing or predicting the session token (session ID cookie). The attacker impersonates the legitimate user without needing their password.

  • How sessions work: After login, server issues Session ID (cookie). Client sends this cookie with every request. Server trusts anyone presenting a valid session ID.
  • Attack methods: (a) Cookie sniffing over HTTP → inject cookie into own browser → instant access. (b) XSS cookie theft → malicious JS sends cookie to attacker. (c) Session fixation → attacker forces victim to use a known session ID. (d) Predictable session IDs → brute-force valid session IDs.
  • Countermeasures: HTTPS everywhere (encrypts cookies). HttpOnly flag (JS cannot read cookie). Secure flag (cookie only sent over HTTPS). Session timeout. Regenerate session ID after login. Bind session to IP/User-Agent.
Password Attacks

Attempts to discover or bypass user passwords to gain unauthorized access.

  • Brute Force: Try every possible combination. 100% effective eventually. Limited by: password length, complexity, lockout policies. Example: 8-char lowercase = 26^8 = 208 billion combinations.
  • Dictionary Attack: Try words from a wordlist (dictionary). Most passwords are English words + numbers. Tool: Hashcat with RockYou wordlist (14 million real passwords from a breach). Much faster than brute force.
  • Rainbow Table Attack: Pre-computed table of hash → password mappings. Given a hash, look up the password in seconds. Defeated by: password salting (adds random value before hashing — makes pre-computed tables useless).
  • Credential Stuffing: Using username/password pairs leaked from one breach to try on other sites. Works because people reuse passwords. Example: LinkedIn breach passwords used to attack Dropbox.
  • Phishing: Trick user into typing their password on a fake login page. No technical hacking needed — social engineering. Countermeasure: MFA, FIDO2 hardware keys (phishing-resistant).

3.5 — Packet Fragmentation Attack
Mnemonic
"Fragments FOOL the Firewall — split the attack to bypass inspection"
Normal: IP splits big packets → Attacker exploits reassembly to evade IDS/crash system
  • Normal IP Fragmentation: When packet size exceeds MTU (Maximum Transmission Unit — typically 1500 bytes), router splits it into fragments. Each fragment has: ID (same for all fragments of same packet), Fragment Offset (position in original packet), More Fragments (MF) flag.
  • Tiny Fragment Attack: Attacker creates an abnormally small first fragment (only 8 bytes — just TCP ports). TCP header split across fragments 1 and 2. Packet filter sees only fragment 1 and cannot determine port/flags → lets it through. Fragment 2 reassembles with malicious content.
  • Teardrop Attack: Sends fragments with overlapping offsets — Fragment 2's offset says it starts before Fragment 1 ends. Reassembly calculation: length = (end of frag2) - (start of frag2) = negative value. Old OS (Win95/NT) crash when calculating negative buffer length. Fully patched in modern systems.
  • Fragment Overlap Attack: Fragment 1 = innocent traffic (passes IDS). Fragment 2 = overlapping, replaces Fragment 1's content with malicious payload. IDS only inspects first fragment → attack slips through. Destination reassembles → malicious complete packet.
  • Countermeasures: Reassemble fragments at firewall/IDS before inspection (resource-intensive but necessary). Drop fragments smaller than minimum (64 bytes). Limit number of fragments per flow. Modern NGFW (Next-Generation Firewall) handles all these cases.
Ref:Stallings Ch.7 "Network Attacks"

4.1 — Authentication as a Countermeasure
Mnemonic — 3 Factors
"KHA — Know it, Have it, Are it"
Something you KNOW (password) · Something you HAVE (token) · Something you ARE (biometric)
Definition

Authentication is the process of verifying the identity of a user, system, or entity. It answers: "Are you really who you claim to be?" It is the first gate of access control. Strong authentication is the most effective countermeasure against unauthorized access.

  • Factor 1 — Something you KNOW: Password, PIN, security question answer. Weakest factor — can be stolen, guessed, or phished. Must enforce: minimum length (12+ chars), complexity, no reuse, no dictionary words.
  • Factor 2 — Something you HAVE: Hardware token (RSA SecurID), smartphone authenticator app (Google Authenticator — TOTP), smart card, USB security key (YubiKey). One-time codes expire in 30 seconds — replay attacks useless.
  • Factor 3 — Something you ARE: Fingerprint, retina scan, face recognition, voice recognition. Biometrics. Cannot be forgotten or easily stolen. Limitation: once compromised, cannot be changed (you only have 10 fingerprints).
  • MFA (Multi-Factor Authentication): Combining 2+ factors. Even if password stolen — attacker still needs physical device. Banks (ATM: card + PIN), Google (password + phone OTP). Reduces account compromise by 99.9% (Microsoft research).
  • Kerberos Protocol: Enterprise authentication — no passwords sent over network. Client gets Ticket Granting Ticket (TGT) from KDC (Key Distribution Centre). Uses TGT to get service tickets for each resource. Used in Windows Active Directory.
  • Digital Certificates / PKI: X.509 certificates bind a public key to an identity, signed by a trusted Certificate Authority (CA). Used in HTTPS — browser verifies website's certificate before trusting it. Foundation of internet security.
Ref:Stallings Ch.14 "Authentication"
4.2 — Encryption as a Countermeasure
Mnemonic
"SAS — Symmetric for Speed, Asymmetric for Sharing keys Safely"
AES (symmetric) = fast bulk encryption · RSA/ECC (asymmetric) = secure key exchange
  • AES-256 (Symmetric): Same key encrypts and decrypts. Extremely fast. Used for: disk encryption (BitLocker, VeraCrypt), VPN tunnels, file encryption. Key problem: how do you share the key securely? Solved by asymmetric encryption.
  • RSA/ECC (Asymmetric): Public key encrypts (share freely), private key decrypts (keep secret). Slow — only used for: key exchange, digital signatures, certificate signing. RSA-2048 minimum; ECC-256 equivalent strength with smaller key.
  • TLS (Transport Layer Security): Combines both. Handshake phase: RSA/ECDHE (asymmetric) exchanges symmetric session key. Data phase: AES-256 (symmetric) encrypts actual data. SHA-256 (hash) ensures integrity. X.509 certificate authenticates server. HTTPS = HTTP + TLS.
  • Hashing (SHA-256, SHA-3): One-way function — input → fixed-size digest. Cannot reverse. Used for: password storage (store hash, not password), file integrity verification, digital signatures, blockchain.
  • End-to-End Encryption (E2EE): Data encrypted at sender, decrypted only at recipient — even service provider cannot read. Used in: WhatsApp (Signal Protocol), Signal app, PGP email. Countermeasure against: server-side breaches, ISP surveillance, government interception.
  • Encryption roles in IW countermeasures: Confidentiality (AES encryption), Integrity (HMAC), Authentication (Digital certificates), Non-repudiation (Digital signatures), Data at rest (Full disk encryption).
Ref:Stallings Ch.2-5 "Cryptography"
4.3 — Auditing and Monitoring
Mnemonic
"AML — Audit Makes Liars accountable"
Auditing = after-the-fact record · Monitoring = real-time watchdog
  • Auditing: Recording security-relevant events in logs for later analysis. Examples: login/logout events, file access, privilege escalation, system changes, network connections. Provides evidence for forensic investigation. Required for compliance (PCI-DSS, HIPAA, SOX).
  • Key Audit Events to capture: All successful and failed login attempts, privileged account usage, changes to security policies, file access to sensitive data, network connection attempts, system startup/shutdown.
  • Monitoring: Real-time analysis of system and network activity. Tools: SIEM (Security Information and Event Management) — collects logs from all devices, correlates events, fires alerts. Example: Splunk, IBM QRadar, Microsoft Sentinel.
  • SIEM Capabilities: Log aggregation (one place for all logs), Event correlation (connect related events across systems — failed login + successful login from different IP = alert), Real-time alerting, Dashboards, Compliance reporting.
  • Why both are needed: Monitoring catches attacks in real-time (minute-level). Auditing provides historical record for investigation (days/weeks later). Together: detect + prove what happened.
  • Limitation: Auditing generates enormous log volume — average enterprise: billions of log events per day. Manual review impossible. SIEM with ML-based correlation is essential to find the needle in the haystack.

4.4 — Firewalls: Types and Architecture
Mnemonic — 5 Firewall Types
"PASTA — Packet filter, Application proxy, Stateful, Transparent, Application-aware (NGFW)"
Each generation adds more intelligence and inspection depth
Definition (Stallings)

A firewall is a security system that monitors and controls incoming and outgoing network traffic based on a set of predefined security rules. It establishes a barrier between trusted internal networks and untrusted external networks.

  • 1. Packet Filtering Firewall (1st Gen): Examines each packet in isolation. Filters based on: Source/Dest IP, Source/Dest Port, Protocol (TCP/UDP/ICMP). Simple ACL rules. No state tracking — cannot tell if a packet is part of an established connection. Fast but limited. Cannot stop application-layer attacks.
  • 2. Stateful Inspection Firewall (2nd Gen): Maintains a state table tracking all active connections. Allows only packets that are part of an ESTABLISHED connection initiated from inside. Can detect TCP sequence number anomalies, SYN flood attacks. Most common enterprise firewall type.
  • 3. Application Proxy / Gateway Firewall: Acts as a relay for specific applications. Full deep inspection at application layer (L7). Example: web proxy examines all HTTP/HTTPS content. Cannot be bypassed by tunnelling attacks (all traffic inspected). Slower — creates two connections for every session.
  • 4. Transparent / Switched Firewall: Operates at Layer 2 (Data Link). No IP address on data path — invisible to network. Insert between two segments without reconfiguring IP scheme. Cannot be targeted by IP-based attacks.
  • 5. Next-Generation Firewall (NGFW): Combines: stateful inspection + application identification (even if port 80) + user identity (who, not just IP) + SSL/TLS inspection + built-in IPS + threat intelligence feeds. Examples: Palo Alto, Fortinet, Cisco Firepower. The modern gold standard.
Limitations of Firewalls

1. Cannot inspect encrypted traffic (unless doing SSL inspection — raises privacy concerns).
2. Cannot stop insider threats (malicious authorized users bypass firewall).
3. Cannot protect against social engineering (phishing bypasses firewall).
4. Application-layer attacks bypass port-based rules (malware over HTTP/80).
5. Zero-day exploits may not have matching signatures yet.
6. VPN/tunnelling can bypass firewall rules.
7. Misconfigured rules leave gaps — human error is the biggest failure point.

Positive vs Negative Filtering

Positive Filtering (Whitelist): ALLOW only explicitly approved traffic. Default: DENY all. More secure. Example: Only allow HTTP(80), HTTPS(443), DNS(53) — block everything else. Used in: high-security environments, banks, military systems.

Negative Filtering (Blacklist): BLOCK only explicitly banned traffic. Default: ALLOW all. Easier to manage. Cannot block unknown threats. Example: Block known malicious IPs, block Telnet(23). Used in: general ISP, consumer products, spam filters.

5.1 — Host-Based IDPS (HIDS)
Mnemonic — What HIDS monitors
"SAFLS — Syscalls, Audit logs, File integrity, Login events, System calls"
Everything happening INSIDE one host machine
Definition

A Host-Based Intrusion Detection System (HIDS) is installed on individual host machines (servers, workstations). It monitors OS-level events, system calls, file activity, user actions, and log files to detect intrusions occurring within that specific host.

  • System Call Monitoring: HIDS intercepts all system calls (open, read, write, exec, fork, socket). Unusual call sequences indicate attack. Example: Web server process making exec() calls → likely PHP webshell being executed by attacker.
  • File Integrity Monitoring (FIM): Takes SHA-256 hash of critical system files (/etc/passwd, /bin/ls, system DLLs). Periodically re-hashes and compares. Any unauthorized change = alert. Catches rootkits that replace system binaries. Tool: Tripwire, AIDE.
  • Log Analysis: Monitors OS logs in real-time. Example: 100 failed SSH logins in 60 seconds = brute force attack → alert + auto-block IP. Windows Event Log analysis — looks for Event ID 4625 (failed login), 4672 (special privileges assigned).
  • Process Monitoring: Tracks all running processes, parent-child relationships. Alert: Apache (web server) spawning cmd.exe or bash shell = likely command injection attack.
  • Registry Monitoring (Windows): Critical registry keys monitored for changes — startup keys, security settings, service configurations. Malware often uses registry for persistence.
  • Network Connection Monitoring: Tracks all network connections made by the host. Alert: a normally isolated database server making outbound connections to the internet = data exfiltration attempt.
✅ HIDS Advantages
  • Sees inside encrypted traffic (already decrypted at host)
  • Detects insider threats
  • Catches local privilege escalation
  • No network tap needed
  • Works even on isolated hosts
❌ HIDS Limitations
  • Only monitors ONE host
  • Agent must be installed on every host
  • Attacker can disable HIDS if they get root
  • High CPU overhead on host
  • Misses network-level attacks
Popular HIDS Tools

OSSEC: Open-source, most widely deployed HIDS. Real-time log analysis, FIM, rootkit detection, active response. Works on Linux, Windows, macOS.
Wazuh: OSSEC fork with modern web GUI (Kibana). Built-in compliance dashboards (PCI-DSS, GDPR, HIPAA). Cloud-native support.
Tripwire: Specializes in FIM. Enterprise-grade. Used by banks and government agencies.

5.2 — Network-Based IDPS (NIDS/NIPS)
Mnemonic — 5 Components
"SCAN-R — Sensor, Console, Analyser, Network tap, Response engine"
5 core components every NIDPS must have
Definition

A Network-Based IDPS monitors and analyses network traffic across an entire network segment. It captures packets from the network, analyses them using signature/anomaly/stateful methods, and generates alerts or blocks attacks. One sensor can protect many hosts simultaneously.

  • Sensor / Monitoring Engine: Captures raw network packets using libpcap (Linux) or WinPcap (Windows). Placed at strategic points: internet edge (sees all external traffic), DMZ, internal network segments. In IPS mode: sits inline and can drop packets in real-time.
  • Detection Engine: The brain. Performs: signature matching (vs rule database), protocol analysis (validates protocol correctness), stateful tracking (maintains connection state), anomaly detection (statistical baseline comparison). Modern engines (Snort3, Suricata) are multi-threaded.
  • Signature / Rule Database: 50,000+ signatures (Snort community). Rules specify: what to look for (content, flags, size), where to look (network, transport, application layer), what to do (alert, drop, log). Updated daily from threat intelligence feeds.
  • Management Console: Web-based dashboard for security analysts. Shows: real-time alert stream, geographic attack maps, top attacker IPs, event trends. Allows rule management, tuning, and reporting. Examples: Splunk, ELK Stack (Elastic, Logstash, Kibana), IBM QRadar.
  • Response Engine: Automated responses: TCP Reset (sends RST to kill connection), Firewall rule insertion (dynamic IP block), SNMP trap (notify network management), Email/SMS alert, Full packet capture (preserve evidence for forensics).
Passive (IDS) vs Inline (IPS) Sensor Architecture

Passive (Promiscuous mode): Gets a COPY of traffic from network SPAN port or tap. CANNOT block in real-time. Zero impact on network latency. No single point of failure. Detection only → IDS mode.

Inline (Bridge mode): Traffic PASSES THROUGH the sensor. Can DROP malicious packets before they reach destination. Adds latency (2-10ms). Must have fail-open/fail-close bypass for hardware failure. Full prevention → IPS mode.

5.3 — Wireless IDPS (WIDS)
Mnemonic
"WIDS Watches Wireless With Sensors Everywhere"
Sensors monitor RF spectrum → detect rogue APs, deauth attacks, WEP use, unauthorized devices
  • What WIDS detects: Rogue access points (unauthorized APs on corporate network), Evil Twin attacks (duplicate SSID), Deauthentication attacks (frame flood), WEP usage (insecure protocol), Ad-hoc networks (device-to-device bypassing network controls), MAC spoofing.
  • Sensor placement: Dedicated WIDS sensors placed throughout the facility to achieve full RF coverage. Overlapping coverage ensures all channels (1-14 for 2.4GHz, 36-165 for 5GHz) are monitored simultaneously.
  • Detection methods: RF fingerprinting (each device has unique radio characteristics), MAC address tracking, SSID monitoring, Channel analysis, Signal strength mapping (locate rogue AP physically using triangulation).
  • Response capabilities: Automated containment — send deauth frames to clients connected to rogue AP. Alert administrator. Block rogue AP's MAC address on network switches. Generate forensic report.
  • Tools: Cisco Adaptive Wireless IPS, Aruba RFProtect, open-source: Kismet (wireless network detector and sniffer). Kismet: detects hidden SSIDs, rogue APs, identifies wireless attack tools in use.

5.4 — Tool Selection and Open-Source IDS Tools
Mnemonic — 5 Key Tools
"SNOSA — Snort, NFR, OSSEC, Suricata, Auditd"
The 5 tools most likely asked in exams for CPISE17
Tool Selection Criteria

When choosing an IDS tool, evaluate: Detection capability (signature + anomaly), Performance (can it handle your network speed?), False positive rate, Ease of management, Update frequency (signature DB), Integration with existing SIEM, Cost (open-source vs commercial), Community support and documentation.

ToolTypeKey FeaturesLimitations
Snort NIDS/NIPS World's most deployed. 50,000+ rules. Real-time traffic analysis, protocol decoding, packet logging. 3 modes: Sniffer, Packet Logger, NIDS. Huge community. Free. Single-threaded (slow on 10Gbps+). No built-in GUI. Cannot inspect TLS. High FP without tuning. Complex rules language.
Suricata NIDS/NIPS Multi-threaded (much faster than Snort). GPU acceleration. Snort rule compatible. Built-in HTTP/TLS/SMB/DNS protocol parsing. JSON output for SIEM. File extraction. Lua scripting. Higher RAM usage. Smaller community. Some Snort rules don't port perfectly. More complex config.
OSSEC/Wazuh HIDS Real-time log analysis. FIM. Rootkit detection. Active response (auto-block). Linux + Windows + macOS. Agent-based. Wazuh adds: Kibana GUI, compliance dashboards, cloud support. HIDS only — no network traffic analysis. Agent on every host. Complex setup. High server resource needs (Wazuh).
Zeek (Bro) Network Monitor Deep protocol analysis + structured logging. Rich logs: HTTP, DNS, FTP, SSL, SMB. Zeek scripting for custom detection. Excellent for forensics and threat hunting. Not a traditional IDS — no real-time alerts by default. Very steep learning curve. Requires SIEM integration. Complex scripting language.
NFR Security NIDS N-code scripting language (flexible detection). Full packet capture. Policy-based specification detection. Good for compliance monitoring. Enterprise sensor management. Primarily commercial. Less active open-source version. Smaller community. Dated compared to Suricata. Vendor updates required.
Snort Architecture — 3 Modes

Mode 1 — Sniffer: Simply reads and prints packets to screen. snort -v
Mode 2 — Packet Logger: Captures packets to disk for later analysis. snort -l /var/log/snort
Mode 3 — NIDS: Full detection mode using rules. snort -c /etc/snort/snort.conf -A console

Snort processes packets through: Preprocessors (defrag, stream reassembly, HTTP decode) → Detection Engine (rule matching) → Alert/Log (output plugins).

NFR (Network Flight Recorder) Security — Exam Focus

NFR was one of the earliest commercial NIDS (founded 1995 by Marcus Ranum). Key concepts for exam:
• Uses N-code language for writing custom detection rules (more powerful than Snort rules)
• Supports specification-based detection (define what IS normal, not just what is bad)
• Full packet capture with indexing for fast forensic retrieval
• Probe/Recorder/Console architecture: Probes capture traffic → Recorders store/analyse → Console manages all
• Acquired by McAfee/Network Associates — now part of larger security suite

Recommended Modern SOC Architecture

Layer 1 (Network): Suricata inline IPS at internet edge → blocks known attacks in real-time
Layer 2 (Host): Wazuh agents on all servers → FIM, log analysis, rootkit detection
Layer 3 (Protocol): Zeek for deep traffic logging → forensics and threat hunting
Layer 4 (Management): Elastic SIEM — correlates all events → unified alert dashboard → SOC analysts
This architecture covers: network attacks + host compromises + protocol anomalies + historical forensics. Defense in Depth!

OS-Level IDS — Exam Points

OS-level IDS (HIDS at OS level) specifically monitors at the operating system kernel:
• System call interception: Every app-to-kernel interaction monitored (auditd, Linux Audit Framework)
• Kernel module IDS: Loaded as kernel module — cannot be killed by user-space attacker without root+skill
• LKM rootkit detection: Detects malicious kernel modules loaded to hide processes/files
• Windows: ETW (Event Tracing for Windows) — high-performance OS-level event logging used by all modern HIDS
• Advantage: Cannot be bypassed by any user-level attack — kernel sees everything
